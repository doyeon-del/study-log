# 01. 핵심 개념

## 워크플로우 = 노드의 그래프

워크플로우는 노드를 선으로 이은 그래프다. 왼쪽 노드의 출력이 오른쪽 노드의 입력이 되고, 데이터는 항상 왼→오로 흐른다. 한 워크플로우에는 보통 트리거 하나로 시작해 여러 액션 노드가 이어진다.

## 노드의 세 가지 역할

- **트리거 노드**: 워크플로우를 *시작*시킨다. 워크플로우마다 보통 하나. "언제 도는가"를 정의.
- **액션(앱) 노드**: 외부 서비스에 뭔가를 한다 — 메일 보내기, 시트에 행 추가, HTTP 요청, Slack 메시지 등.
- **로직/코어 노드**: 데이터를 가공·분기·반복한다 — IF, Switch, Merge, Set(=Edit Fields), Code, Loop, Filter.

## 트리거의 종류 (언제 도는가)

| 트리거 | 언제 |
| --- | --- |
| Manual | 캔버스에서 직접 실행 (테스트용) |
| Schedule | cron 일정 — 매분/매시/매일/커스텀 |
| Webhook | 외부에서 이 URL로 요청이 오면 |
| Form | n8n이 띄운 폼이 제출되면 |
| Chat Trigger | 채팅 입력이 오면 (AI 에이전트용) |
| 앱 트리거 | Gmail 새 메일, Notion 새 페이지 등 서비스 이벤트 |

## 실행 모델

- **실행(execution)** = 워크플로우 한 번 도는 것. 노드가 몇 개든 한 번 도는 게 1 실행.
- n8n Cloud는 이 실행 횟수로 과금한다. 셀프호스팅은 실행 무제한(서버 자원이 한계).
- 실행마다 로그가 남아, 각 노드에서 어떤 JSON이 들고 났는지 사후에 그대로 볼 수 있다. 디버깅의 기본.

## 두 가지 모드

- **에디터(수동 테스트)**: 노드를 하나씩 눌러 실행하며 출력 JSON을 확인. 만드는 중엔 여기서 산다.
- **프로덕션(active)**: 워크플로우를 켜두면 트리거가 알아서 돈다(스케줄/웹훅 등).

## 알아두면 편한 기능

- **Pinned data**: 노드 출력을 고정해, 외부 API를 다시 때리지 않고 뒤 노드만 반복 테스트.
- **Sub-workflow**: 워크플로우를 함수처럼 떼어내 다른 워크플로우에서 호출. 길어지면 쪼갠다.
- **Credentials**: API 키·토큰은 노드가 아니라 별도 자격증명으로 관리되고 암호화 저장된다.
